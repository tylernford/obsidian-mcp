# Architecture

## Overview

This is an MCP (Model Context Protocol) server that bridges Claude Code to an Obsidian vault. It translates MCP tool calls into HTTP requests against the [Local REST API](https://github.com/coddingtonbear/obsidian-local-rest-api) Obsidian plugin.

```
Claude Code <──stdio──> MCP Server <──HTTP──> Local REST API plugin <──> Obsidian
```

The server runs as a Node.js process, communicating with Claude Code over stdin/stdout via the MCP SDK's `StdioServerTransport`, and with Obsidian over localhost HTTP using the native `fetch` API.

## Project Structure

```
obsidian-mcp/
├── src/
│   ├── index.ts          # Entry point — wires up client, server, and tools
│   ├── api-client.ts     # HTTP client for the Local REST API
│   └── tools/
│       ├── vault.ts      # CRUD operations on vault files
│       ├── search.ts     # Full-text and Dataview DQL search
│       ├── metadata.ts   # Tag and frontmatter management
│       ├── commands.ts   # Obsidian command discovery and execution
│       ├── active-file.ts# Read/update the currently open note
│       ├── navigation.ts # Open files in the Obsidian UI
│       └── periodic.ts   # Read/update periodic notes (daily, weekly, etc.)
├── docs/                 # Design docs, plans, and backlog
└── package.json
```

## Entry Point

`src/index.ts` is the sole entry point. It:

1. Instantiates `ObsidianClient` with credentials from environment variables
2. Creates an `McpServer` (from `@modelcontextprotocol/sdk`)
3. Calls each `register*Tools(server, client)` function to register all 15 tools
4. Connects the server to a `StdioServerTransport`

There is no routing, middleware, or framework — the MCP SDK handles message framing and dispatch.

## API Client

`src/api-client.ts` exports the `ObsidianClient` class, which is the only component that makes HTTP requests. All tool handlers go through this client rather than calling `fetch` directly.

### Key methods

- **`request(method, path, options)`** — General-purpose HTTP method. Handles Bearer auth, query parameters, content-type negotiation (JSON vs plain text), and connection errors (e.g. `ECONNREFUSED` when Obsidian isn't running).
- **`patch(path, options)`** — Specialized method for the REST API's PATCH semantics, which use custom headers (`Operation`, `Target-Type`, `Target`, `Create-Target-If-Missing`) to apply targeted updates at a heading, block reference, or frontmatter field.
- **`encodePath(path)`** — Per-segment URL encoding for vault file paths.

### Response shape

All client methods return a normalized result object:

```js
{ ok: true,  status: number, data: any }   // success
{ ok: false, status: number, error: string } // failure
```

Tool handlers check `result.ok` and either return the data or surface the error to the LLM.

## Tool Modules

Each file in `src/tools/*.ts` exports a single `register*Tools(server, client)` function. These functions call `server.tool()` to register one or more MCP tools with:

- A **name** (e.g. `vault_read`)
- A **description** (shown to the LLM for tool selection)
- A **Zod schema** defining the tool's input parameters
- An **async handler** that calls `client.request()` or `client.patch()` and returns an MCP content response

### Tool inventory

| Module           | Tools                                                                      | REST API endpoints                       |
| ---------------- | -------------------------------------------------------------------------- | ---------------------------------------- |
| `vault.ts`       | `vault_list`, `vault_read`, `vault_create`, `vault_update`, `vault_delete` | `GET/PUT/PATCH/DELETE /vault/{path}`     |
| `search.ts`      | `search`                                                                   | `POST /search/simple/`, `POST /search/`  |
| `metadata.ts`    | `tags_manage`, `frontmatter_manage`                                        | `GET/PATCH /vault/{path}` (JSON format)  |
| `commands.ts`    | `commands_list`, `commands_execute`                                        | `GET /commands/`, `POST /commands/{id}/` |
| `active-file.ts` | `active_file_read`, `active_file_update`                                   | `GET/PATCH /active/`                     |
| `navigation.ts`  | `file_open`                                                                | `POST /open/{path}`                      |
| `periodic.ts`    | `periodic_read`, `periodic_update`                                         | `GET/POST/PATCH /periodic/{period}/`     |

### Response convention

All tool handlers return the same MCP content shape:

```js
// Success
{ content: [{ type: "text", text: "..." }] }

// Error
{ content: [{ type: "text", text: "error message" }], isError: true }
```

### Notable patterns

- **Content negotiation** — `vault_read`, `active_file_read`, and `periodic_read` accept a `format` parameter (`"markdown"` or `"json"`) that maps to an `Accept` header (`text/markdown` vs `application/vnd.olrapi.note+json`).
- **Targeted PATCH updates** — `vault_update`, `active_file_update`, and `periodic_update` share the same update model: an `operation` (append/prepend/replace) applied at a `targetType` (heading/block/frontmatter) identified by a `target` string. These all flow through `client.patch()`.
- **Auto-creation on 404** — `periodic_update` retries after a 404 by POSTing to create the periodic note from its template, then re-attempting the PATCH.
- **Read-modify-write for tags** — `tags_manage` fetches the current tag array, applies additions/removals in memory, then patches the full array back.

## Dependencies

| Package                     | Purpose                                                                     |
| --------------------------- | --------------------------------------------------------------------------- |
| `@modelcontextprotocol/sdk` | MCP server framework (stdio transport, tool registration, message handling) |
| `zod`                       | Schema definition and input validation for tool parameters                  |

No other runtime dependencies. HTTP is handled by Node's built-in `fetch` (requires Node 18+).
