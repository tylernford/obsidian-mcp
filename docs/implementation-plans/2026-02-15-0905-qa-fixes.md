## Tasks

### Task 1: Harden `api-client.ts`

**Description:** Make `ApiResponse` generic, export key types, widen `apiKey` to `string | undefined`, wrap `JSON.parse` in try/catch, and fix the `content-length: 0` early return to check `response.ok` first. This is the foundation task — Tasks 2 and 3 depend on the type changes here.

**Files:**

- `src/api-client.ts` — modify

**Changes:**

1. Make `ApiResponse` generic:

```ts
// Before
type ApiResponse =
  | { ok: true; status: number; data: unknown }
  | { ok: false; status: number; error: string };

// After
export type ApiResponse<T = unknown> =
  | { ok: true; status: number; data: T }
  | { ok: false; status: number; error: string };
```

2. Export `ObsidianClientConfig` and widen `apiKey`:

```ts
// Before
interface ObsidianClientConfig {
  apiKey: string;

// After
export interface ObsidianClientConfig {
  apiKey: string | undefined;
```

3. Wrap `JSON.parse` in try/catch (around line 107):

```ts
// Before
} else if (contentType.includes("json")) {
  data = JSON.parse(text);
}

// After
} else if (contentType.includes("json")) {
  try {
    data = JSON.parse(text);
  } catch {
    return {
      ok: false,
      status: response.status,
      error: `Malformed JSON response from Obsidian API: ${text.slice(0, 200)}`,
    };
  }
}
```

4. Fix `content-length: 0` to check `response.ok` first (around line 97):

```ts
// Before
if (response.status === 204 || contentLength === "0") {
  return { ok: true, status: response.status, data: null };
}

// After
if (response.status === 204 || (contentLength === "0" && response.ok)) {
  return { ok: true, status: response.status, data: null };
}
```

5. Update `patch()` and `request()` return types to `Promise<ApiResponse>` (already compatible since `ApiResponse<T = unknown>` defaults).

**Done when:**

- `ApiResponse<T>` is generic and exported
- `ObsidianClientConfig` is exported with `apiKey: string | undefined`
- `JSON.parse` wrapped in try/catch
- `content-length: 0` with non-ok status falls through to error handling
- `pnpm typecheck` passes

**Commit:** "Harden api-client with generic ApiResponse, safe JSON parsing, and type exports"

### Task 3: Fix `metadata.ts` — validation guards and generic types

**Description:** Add runtime validation for `tags` (in `tags_manage`) and `key`/`value` (in `frontmatter_manage`) before the code uses them. Replace `as` casts with generic `ApiResponse<T>` type parameters. Remove all `!` non-null assertions.

**Depends on:** Task 1

**Files:**

- `src/tools/metadata.ts` — modify

**Changes:**

1. Add validation guard for `tags` before use (after the `list` early return, before line 53):

```ts
if (!tags || tags.length === 0) {
  return {
    content: [
      { type: "text", text: "tags are required for add/remove actions" },
    ],
    isError: true,
  };
}
```

2. Add validation guard for `key`/`value` before use (before line 135):

```ts
if (!key || !value) {
  return {
    content: [
      { type: "text", text: "key and value are required for the set action" },
    ],
    isError: true,
  };
}
```

3. Replace `as` casts with generic type parameters:

```ts
// Before (line 41)
const noteData = readResult.data as { tags?: string[] };

// After — use generic on the ok check
// Since we've already checked readResult.ok, narrow via the discriminated union:
const noteData = readResult.data as { tags?: string[] };
// Actually, to use the generic, we'd type the request call. But request() returns
// ApiResponse<unknown> by default. The cleaner approach: since we've already narrowed
// with the ok check, cast the data field. The design spec says "use generic type parameter
// instead of as casts." We can type the variable after the ok-narrowing:
```

Actually, the cleanest approach given the current `request()` signature returns `ApiResponse` (generic defaults to `unknown`): the `as` casts on `.data` after an `ok` check are the idiomatic pattern. The design spec decision #4 says to update call sites that have `as` casts to use the generic. To do this, we make the response variable typed:

```ts
// Before
const readResult = await client.request("GET", `/vault/${encodedPath}`, { ... });
// ...
const noteData = readResult.data as { tags?: string[] };

// After
const readResult = await client.request("GET", `/vault/${encodedPath}`, { ... }) as ApiResponse<{ tags?: string[] }>;
// ...after ok check, readResult.data is typed as { tags?: string[] }
if (!readResult.ok) { ... }
const currentTags = readResult.data.tags || [];
```

Same pattern for the `frontmatter_manage` read path.

4. Remove `tags!` (line 55) — now safe after validation guard.
5. Remove `key!` and `value!` (lines 138-139) — now safe after validation guard.

**Done when:**

- Zero `!` assertions in `src/tools/metadata.ts`
- `tags_manage` returns error when `action` is `add`/`remove` and `tags` is missing/empty
- `frontmatter_manage` returns error when `action` is `set` and `key`/`value` is missing
- `as` casts replaced with generic type parameters
- `pnpm typecheck` passes

**Commit:** "Add runtime validation guards and use generic ApiResponse in metadata tools"
